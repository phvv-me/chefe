    Manifest¶

    一切都存在一份 chefe.toml 里。头部配置 workspace，[deps] 和各language/toolchain专属的表项声明包，平台叠加层和具名环境则在其上层层组合。

    Workspace¶

    [workspace]
name     = "my-project"
platforms = ["osx-arm64", "linux-64"]
channels = ["conda-forge", "nvidia"]
dotenv   = true                 # read .env into the environment, on by default

    依赖¶

    裸 [deps] 表项就是 conda，即默认 resolver。裸字符串是一条版本规格说明，* 表示任意版本。

    [deps]
python = ">=3.11"
nodejs = ">=22"
ripgrep = "*"
pueue  = ">=4"

    Python¶

    Python 包由 pixi 解析进同一个环境。[python] 存放设置，[python.deps] 存放包，[python.indexes] 存放具名的额外索引。

    [python]
index-strategy = "unsafe-best-match"

[python.indexes]
pytorch = "https://download.pytorch.org/whl/cu124"

[python.deps]
torch = { version = ">=2.6", index = "pytorch" }
ruff  = ">=0.6"

    Runtime 键控的 toolchain¶

    其他每个language/toolchain都是显式声明的，不做任何推断。

    [rust.deps]
bookokrat = "*"

[nodejs.dev.deps]
prettier = ">=3"
"@tobilu/qmd" = "*"

    运行时会被自动保证

    每个language/toolchain都需要它的语言运行时才能安装和运行，所以当缺失时 chefe 会从 conda-forge 把它补上。[python.deps] 保证 python。 nodejs、rust 等 toolchain 应在 [deps] 中声明。如果你在 [deps] 里自己声明了 钉死版本的 python（或其他），chefe 会原样保留它。这在每个环境里同样成立， 所以一个使用了 [python.deps] 的 no-default 环境照样会拿到它自己的 python。

    选择 npm 驱动¶

    Node.js language/toolchain使用 npm 注册表，[nodejs] 中的 manager 指定安装它的二进制。无论项目选择哪一个，编译出的 package.json 都相同；chefe 会在生成的环境目录内运行所指定的工具，因此任何会安装到其工作目录的 包管理器都能工作，哪怕是 chefe 从未听说过的。

    [nodejs]
manager = "pnpm"   # 默认 npm；也可是 pnpm、yarn 或任何兼容二进制

[nodejs.deps]
svelte = ">=5"

    默认是 npm，所以未定义 manager 时 [nodejs.deps] 会用 npm 安装。指定别的工具只改变 chefe 运行的二进制， 依赖、注册表和文件都不变。

    驱动必须在 PATH 上

    在 [deps] 中声明 nodejs；管理器本体由你提供。如果 manager 指向不存在的工具，真实命令运行时会失败。

    JavaScript 应用¶

    默认情况下 [nodejs.deps] 会作为工具安装在 .chefe/ 内，与 conda 环境为邻。应用设置 app = true， chefe 便会安装到项目根目录，并在那里写出完整的 package.json，让 Vite、SvelteKit 等照常解析 node_modules。

    [nodejs]
manager = "pnpm"
app = true

[nodejs.deps]
svelte = ">=5"
vite = ">=8"

[nodejs.package]
type = "module"
pnpm = { onlyBuiltDependencies = ["esbuild", "workerd"] }

    [nodejs.package] 会原样合并进 package.json，因此工具期望的任何字段都会原封不动地通过，从 type、 engines 到 pnpm 的 onlyBuiltDependencies 这类包管理器自有设置。文件由 chefe 写出，所以你只需 编辑 chefe.toml，而生成的 package.json 是可加入 gitignore 的构建产物。

    开发依赖¶

    [dev.*] 镜像基础 scope，用于构建和测试需要、但运行时不需要的工具。每个分组都会编译到其language/toolchain自有的 开发机制，chefe install 默认就会安装它们。

    [dev.deps]            # conda 开发工具
ruff = "*"

[dev.python.deps]       # Python 开发工具
pytest = ">=8"

[nodejs.dev.deps]     # -> package.json 的 devDependencies
vite = ">=8"

    [nodejs.dev.deps] 进入 devDependencies，而 [dev.deps] 和 [dev.python.deps] 会成为加入默认环境的 dev feature，因此你的 linter 和测试运行器会与运行时依赖一起安装。这比完整的 [envs.dev] 更轻量， 后者是一个有自己求解过程的独立环境。

    系统要求¶

    用于跨平台求解的 conda 虚拟包下限，而非模块加载（module-load）。

    [system]
cuda = "13.0"

    环境变量¶

    环境激活时应用的静态变量。.env 也会被默认读取，可用头部的 dotenv 开关切换。

    [env]
LOG_LEVEL = "info"
CUDA_MODULE_LOADING = "LAZY"

    激活脚本¶

    环境激活时被 source 的 shell 脚本，用于静态环境变量无法表达的配置（计算得到的路径、库的符号链接）。它们会编译成 pixi 的 [activation] scripts；用仓库根目录的相对路径就能在生成的 .chefe/ 中继续生效。

    [activation]
scripts = ["scripts/activate.sh"]

    平台叠加层¶

    按平台有条件地添加依赖，它们会编译成原生的 pixi target。任意作用域都可嵌套在 [on.…] 之下。

    python = "*"

[on.linux-aarch64.python.deps]
some-arm-wheel = "*"

    具名环境¶

    像 pixi feature 一样组合出额外的环境。no-default = true 会排除基础依赖，platforms 把环境限制在它能够构建的平台上（这样在笔记本上求解时会跳过一个 GPU 环境）。

    python = "*"

[envs.serving.python.deps]
vllm = ">=0.6"

    用 chefe install serving 或 chefe tree serving 来安装或查看其中之一。

    任务¶

    在环境内运行的具名命令，通过 chefe run <task> 触发。它们运行的是代码，绝不安装依赖。每个包都该归在它所属language/toolchain的 [<eco>.deps] 里，这样 chefe install 和 chefe global 才能管控它。一个跑去执行 npm install -g 或 cargo install 的任务，正是 chefe 要取代的反模式。

    [tasks]
serve = { run = "python -m my_project.server", depends = ["build"] }
build = "python -m my_project.build"
